## 1. API Design Principles

### Core Principles

1. **Idempotency is Critical** - Because the frontend queues logs while offline, every log request must include a `client_log_id` UUID generated on the device. The backend uses this to prevent duplicate entries.

2. **Authoritative Timestamps** - The server must respect the `log_date` generated by the client at time of typing, not the server receipt time. `log_ts` is the server timestamp used only for time-series partitioning.

3. **Parallel Responses** - Submitting a log triggers both a database write (structured data) and an AI agent call (conversational reply + micro-hook). These happen in parallel and are merged into a single response.

4. **Empty-state conventions** - `GET /goals/active`, `GET /experiments/active`, `GET /wraps/latest` return `200 OK` with a `null` object (never `404`) during early days before data exists.

5. **Phase-aware responses** - All progress reporting and pattern framing must reference the user's current phase context.

6. **Goal-agnostic design** - No endpoint logic should assume "fat-loss" values. All goal types must be supported.

- Non-breaking additions can be made to `v1` without version bump

## 3. Authentication & Authorization

### Authentication Method

**OTP (One-Time Password) via SMS**

#### Authentication Flow

1. User enters phone number
2. `POST /auth/send-otp` → Backend sends 6-digit OTP via SMS
3. User enters OTP code
4. `POST /auth/verify-otp` → Backend verifies code, returns JWT token
5. Frontend stores JWT in secure storage (localStorage for web, secure keychain for mobile)
6. All subsequent requests include JWT in `Authorization` header

### JWT Token Structure

{
  "user_id": "uuid",
  "phone_number": "+919876543210",
  "iat": 1709876543,
  "exp": 1712468543
}

**Token expiry:** 30 days  
**Refresh strategy:** MVP uses long-lived tokens. Post-MVP: implement refresh token flow.

### Authorization Header

Authorization: Bearer <jwt_token>

All authenticated endpoints require this header. If missing or invalid:

{
  "error": {
    "code": "UNAUTHORIZED",
    "message": "Invalid or missing authentication token"
  }
}

**HTTP Status:** `401 Unauthorized`

## 5. Error Handling

### HTTP Status Codes

| Status Code | Usage |
|-------------|-------|
| `200 OK` | Successful request |
| `201 Created` | Resource successfully created |
| `400 Bad Request` | Invalid request payload or parameters |
| `401 Unauthorized` | Missing or invalid authentication token |
| `403 Forbidden` | Authenticated but not authorized for resource |
| `404 Not Found` | Resource does not exist |
| `409 Conflict` | Duplicate resource (e.g., duplicate phone number) |
| `422 Unprocessable Entity` | Validation error |
| `429 Too Many Requests` | Rate limit exceeded |
| `500 Internal Server Error` | Server-side error |
| `503 Service Unavailable` | Service temporarily unavailable |

### Error Code Catalog

| Error Code | HTTP Status | Description |
|------------|-------------|-------------|
| `UNAUTHORIZED` | 401 | Invalid or missing auth token |
| `INVALID_OTP` | 400 | OTP code is incorrect |
| `OTP_EXPIRED` | 400 | OTP code has expired (5 min TTL) |
| `PHONE_ALREADY_EXISTS` | 409 | Phone number already registered |
| `VALIDATION_ERROR` | 422 | Request payload failed validation |
| `RESOURCE_NOT_FOUND` | 404 | Requested resource does not exist |
| `RATE_LIMIT_EXCEEDED` | 429 | Too many requests |
| `IDEMPOTENCY_CONFLICT` | 409 | Client log ID already processed with different data |
| `INTERNAL_ERROR` | 500 | Unexpected server error |
| `AI_SERVICE_ERROR` | 500 | LangGraph/AI orchestrator error |

## 6. Rate Limiting

### Rate Limit Rules

| Endpoint Pattern | Limit | Window |
|------------------|-------|--------|
| `POST /auth/send-otp` | 5 requests | Per phone number, per hour |
| `POST /auth/verify-otp` | 10 requests | Per phone number, per hour |
| `POST /logs` | 100 requests | Per user, per hour |
| `POST /photos` | 20 requests | Per user, per hour |
| `GET *` | 300 requests | Per user, per hour |
| `POST /chat` | 60 requests | Per user, per hour |

### Rate Limit Response Headers

X-RateLimit-Limit: 100
X-RateLimit-Remaining: 87
X-RateLimit-Reset: 1709880000

### Rate Limit Exceeded Response

{
  "error": {
    "code": "RATE_LIMIT_EXCEEDED",
    "message": "Too many requests. Please try again in 45 minutes.",
    "details": {
      "retry_after": 2700
    }
  }
}

**HTTP Status:** `429 Too Many Requests`

## 7. Endpoint Specifications

### 7.1 Authentication Endpoints

#### POST /auth/send-otp

**Description:** Send OTP code to phone number via SMS.

**Authentication:** None (public endpoint)

**Request:**

{
  "phone_number": "+919876543210"
}

**Validation:**
- `phone_number` must be valid E.164 format
- Must include country code

**Response (Success):**

{
  "data": {
    "message": "OTP sent to +919876543210",
    "expires_in": 300
  }
}

**HTTP Status:** `200 OK`

**Response (Error - Rate Limited):**

{
  "error": {
    "code": "RATE_LIMIT_EXCEEDED",
    "message": "Too many OTP requests. Please try again in 55 minutes."
  }
}

**HTTP Status:** `429 Too Many Requests`

### 7.3 Logging Endpoints

#### POST /logs

**Description:** Submit a log entry (weight, food, context, or photo). Triggers parallel DB write and AI response generation.

**Authentication:** Required

**Request:**

{
  "client_log_id": "uuid-generated-on-device",
  "log_type": "weight",
  "log_date": "2026-03-08",
  "data": {
    "weight_kg": 91.5
  }
}

**Log type-specific data formats:**

**Weight log:**
{
  "log_type": "weight",
  "log_date": "2026-03-08",
  "data": {
    "weight_kg": 91.5
  }
}

**Food log:**
{
  "log_type": "food",
  "log_date": "2026-03-08",
  "data": {
    "food_description": "Big Mac meal with large Coke",
    "food_source": "manual"
  }
}

**Context log:**
{
  "log_type": "context",
  "log_date": "2026-03-08",
  "data": {
    "context_tags": ["stress_high", "alcohol"],
    "context_note": "Work deadline stress, had 2 beers"
  }
}

**Photo log:**
{
  "log_type": "photo",
  "log_date": "2026-03-08",
  "data": {
    "photo_id": "uuid-from-POST-photos"
  }
}

**Validation Rules:**
- `client_log_id` is required and must be a valid UUID
- `log_date` must not be in the future
- `log_date` can be up to 30 days in the past (retro-log window)
- `weight_kg` must be between 20.0 and 500.0 if provided
- Multiple logs of different `log_type` may share the same `log_date` (valid)

**Response (Success):**

{
  "data": {
    "log": {
      "log_id": "uuid",
      "user_id": "uuid",
      "client_log_id": "uuid-generated-on-device",
      "log_type": "weight",
      "log_date": "2026-03-08",
      "log_ts": "2026-03-08T15:30:00Z",
      "is_retro": false,
      "data": {
        "weight_kg": 91.5
      },
      "phase_index_at_log": 0,
      "created_at": "2026-03-08T15:30:00Z"
    },
    "ai_response": {
      "confirmation": "Logged 91.5kg",
      "micro_hook": "7-day average is down 0.4kg. Exactly on Phase 1 pace.",
      "outlier_context_prompt": null
    }
  }
}

**HTTP Status:** `201 Created`

**Response (Weight Outlier with Context Prompt):**

{
  "data": {
    "log": { ... },
    "ai_response": {
      "confirmation": "Logged 94.0kg",
      "micro_hook": "Thats up 1.8kg from yesterday. Keep watching.",
      "outlier_context_prompt": "Anything different yesterday? Travel, big meal, poor sleep?"
    }
  }
}

**Response (Idempotency - Already Processed):**

{
  "data": {
    "log": {
      "log_id": "uuid-original",
      ...
    },
    "ai_response": {
      ...
    },
    "idempotency_note": "This log was already processed"
  }
}

**HTTP Status:** `200 OK` (not `201` for idempotent requests)

**Response (Error - Validation):**

{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Invalid log data",
    "details": {
      "weight_kg": "Must be between 20.0 and 500.0",
      "log_date": "Cannot be in the future"
    }
  }
}

**HTTP Status:** `422 Unprocessable Entity`

**Response (Error - Missing client_log_id):**

{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "client_log_id is required for idempotency"
  }
}

**HTTP Status:** `400 Bad Request`

## 9. Idempotency Implementation

### Client-Side Requirements

1. **Generate UUID on device** - Every log submission must include a `client_log_id` UUID generated on the device at the time of user input.

2. **Store UUID locally** - The UUID is stored in IndexedDB with the queued log.

3. **Retry with same UUID** - If the request fails (network error, timeout), the frontend retries with the same `client_log_id`.

### Backend-Side Requirements

1. **Check idempotency table** - On every `POST /logs` request, the backend checks the `idempotency_keys` table for the `client_log_id`.

2. **If found (status = "processed")** - Return `200 OK` with the original log response. No duplicate write.

3. **If not found** - Insert key as "pending", proceed with write, update status to "processed".

4. **If conflict (different data)** - Return `409 Conflict` if the same `client_log_id` was submitted with different log data.

### Idempotency Key Expiry

- **TTL:** 72 hours
- **Cleanup:** Hourly cron job deletes expired keys

### Example Flow

**First request:**
POST /logs
{
  "client_log_id": "abc-123",
  "log_type": "weight",
  "log_date": "2026-03-08",
  "data": { "weight_kg": 91.5 }
}

**Response:** `201 Created` (log written to DB)

**Retry (network error):**
POST /logs
{
  "client_log_id": "abc-123",
  "log_type": "weight",
  "log_date": "2026-03-08",
  "data": { "weight_kg": 91.5 }
}

**Response:** `200 OK` (no duplicate write, original log returned)

## 10. Offline-First Considerations

### Frontend Offline Queue

1. **IndexedDB queue** - All log submissions are queued locally in IndexedDB before sending to server.

2. **Background sync** - Service worker automatically syncs queued logs when connection is restored.

3. **Message status indicators** - User sees: "Logging..." → "Logged" → "Synced"

### Backend Offline-Tolerant Design

1. **Accept retro-logs** - `log_date` can be up to 30 days in the past. `is_retro` flag is set automatically.

2. **Respect client timestamps** - `log_date` (client-generated) is the authoritative display date, not server receipt time.

3. **Idempotency** - Prevents duplicate writes when offline queue retries.

### Sync Conflict Resolution

**No conflicts in MVP.** Logs are append-only, immutable. No concurrent editing of the same log from multiple devices.
